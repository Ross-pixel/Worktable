# Worktable — Development Rules

## 1. Перед началом любого модуля

Обязательно ответить:

1. Какую проблему решает модуль?
2. Какие данные он хранит?
3. Может ли он быть удалён без последствий?
4. Нарушает ли он архитектурные инварианты?

Если ответ неясен — модуль не разрабатывается.

---

## 2. Код-правила

- Никакой бизнес-логики в routes.
- Никакой прямой работы с БД вне repository.
- Никаких импортов между модулями.
- Только dependency injection.

---

## 3. Изменения архитектуры

Любое изменение:

1. Должно быть отражено в ARCHITECTURE.md.
2. Не должно нарушать существующие модули.
3. Должно проходить через архитектурный фильтр:
   - Увеличивает ли гибкость?
   - Уменьшает ли связанность?
   - Упрощает ли добавление нового модуля?

Если нет — изменение отклоняется.

---

## 4. Версионирование

Разработка идёт по версиям:

v0.1 — Core + Tasks  
v0.2 — Auth  
v0.3 — User configuration  
v0.4 — Dynamic UI  
v1.0 — SaaS readiness  

Никаких прыжков между версиями.

---

## 5. Главный принцип

Worktable — это система.

Если решение упрощает код,
но усложняет систему — оно неверное.
